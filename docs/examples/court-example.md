# Court Review Example

## Prompt

```bash
/court-review "Review whether Agora v2 is ready for a public positioning update"
```

## Example output

```markdown
## Court Review

### Strategist
- Repositioning Agora as a supervised judgment overlay lowers adoption friction and aligns the product with how teams actually use host agents.
- The shift is strategically sound because it moves differentiation from visible debate theater to durable decision artifacts.

### Operator
- The minimum viable command surface is now present: `/clarify`, `/decide`, `/doubt`, `/minority-report`, `/court-review`, and `/agora`.
- Installation and first-run understanding still depend on concise host-specific onboarding and concrete examples.
- A public push is operationally safer if README quick start, examples, and install docs stay tightly aligned.

### Censor
- Do not imply that all hosts have identical maturity.
- Do not overstate MCP or cross-host support as core requirements.
- Avoid language that makes Agora sound like a replacement runtime rather than an overlay.
- Any launch copy that says "multi-agent" should be checked against the new product thesis so old framing does not re-dominate.

### Historian
- Agora's original strength came from philosophy as a real method, not as roleplay.
- The risk is not losing the old harness identity, but keeping too much of it in the front door and recreating the old adoption friction.
- The transition is strongest when small commands serve as the entrypoint and deeper overlays preserve the project's conceptual core.

### Sovereign Verdict
- Approve with conditions
- Conditions:
  - keep host-specific quick start concise and visible
  - maintain examples for dialectic, genealogy, skeptic, and court modes
  - avoid product copy that restores dialectic as the default for all tasks
- Owner: repo maintainer
```

## Second prompt

```bash
/court-review "Should we allow autonomous agents to publish release notes without human approval?"
```

## Why this matters

Court review is not a generic summary tool.
It is for separating strategy, operation, oversight, precedent, and final responsibility before approval.
